# 电信CRM客户中心微服务系统 — 项目汇报

---

## 一、项目背景

电信行业客户中心承载着客户开户、销户、换卡、资料变更等核心业务，日均业务量达 **300万笔**，峰值 **QPS 500+**。传统单体架构在高并发、跨区域协同等场景下面临瓶颈，亟需向微服务架构演进。

本项目基于 **领域驱动设计（DDD）** 方法论，完成了电信CRM客户中心的全链路微服务设计方案，涵盖业务建模、领域模型、流程设计、接口设计、数据持久化及高并发攻关六大模块。

---

## 二、整体架构

系统拆分为 **8个核心微服务**，通过 API 网关统一接入，服务间通过 OpenFeign/Dubbo 通信：

```
                    ┌──────────────────┐
                    │   API 网关        │
                    └────────┬─────────┘
        ┌───────────┬────────┼────────┬───────────┐
   ┌────▼───┐ ┌────▼───┐ ┌──▼─────┐ ┌▼───────┐ ┌─▼──────┐
   │客户管理│ │用户管理│ │账户管理│ │合约管理│ │权限控制│
   └────────┘ └────────┘ └────────┘ └────────┘ └────────┘
        ┌───────────┬──────────────────┐
   ┌────▼───┐ ┌────▼───┐  ┌───────────▼──┐
   │日志监控│ │基础信息│  │跨中心协同服务│
   └────────┘ └────────┘  └──────────────┘
```

**推荐技术栈：** Spring Cloud Alibaba + Nacos + Gateway + Sentinel + MySQL（主从）+ Redis Cluster + ShardingSphere + Seata + RabbitMQ/Kafka

## 四、设计亮点

### 1. DDD驱动的精准服务拆分

以限界上下文为边界，将电信CRM业务精准拆分为8个职责单一的微服务，每个服务拥有独立数据库，实现**数据隔离**与**独立演进**，避免了"分布式单体"的常见问题。

### 2. 三级缓存架构

设计了 **Nginx → 网关 → 服务** 三级缓存体系，结合 Redis Cluster（分布式缓存）与 Caffeine（本地缓存），在高并发场景下大幅降低数据库压力，将热点数据访问延迟控制在毫秒级。

### 3. 灵活的分布式事务策略

根据业务一致性要求，**按场景选型**：
- **Saga模式**：适用于开户等长流程（最终一致性）
- **TCC模式**：适用于资金操作等短事务（强一致性）
- **本地消息表**：适用于跨服务异步通知

### 4. 跨中心协同机制

专门设计了**跨中心协同微服务**，解决多区域CRM数据同步难题，包含同步任务调度、冲突检测与自动/手动解决策略，保障全网数据一致性。

### 5. 全栈性能优化体系

从 JVM 调优（G1 GC）、数据库优化（索引+分库分表）、缓存策略、到代码级优化（批量操作、异步化），形成了系统化的性能保障方案。

## 五、重难点分析

### 重难点1：日300万业务量的高并发适配

**挑战：** 电信业务存在明显的早高峰（8:00-10:00）和晚高峰（18:00-21:00），峰值流量是均值的3-5倍。

**解决思路：**
- 分层限流：从 Nginx → Gateway → Sentinel 逐层控制流量
- 读写分离 + ShardingSphere 分库分表，水平扩展数据库能力
- 热点数据预加载 + 三级缓存，减少90%以上的数据库读请求
- 异步化非核心链路（日志、通知等），缩短主流程响应时间

### 重难点2：分布式事务的数据一致性

**挑战：** 微服务拆分后，跨服务业务操作（如开户涉及客户、用户、账户三个服务）无法使用传统事务。

**解决思路：**
- Saga编排引擎管理长事务流程，每步操作定义补偿逻辑
- TCC模式确保资金类操作的强一致性（Try-Confirm-Cancel）
- 事务状态持久化 + 定时巡检，自动恢复异常中断的事务

### 重难点3：跨区域数据同步与冲突解决

**挑战：** 多地市CRM中心的数据需要实时同步，网络延迟和并发修改可能导致数据冲突。

**解决思路：**
- 基于消息队列的异步同步机制，解耦各中心依赖
- 版本号 + 时间戳的冲突检测策略
- 自动合并规则 + 人工审核兜底的冲突解决流程
